# AICreator — Платформа AI-генерации кода

## Контекст

Корпоративный архитектор крупной ритейл-компании проектирует техническое решение для ускорения работы команд системной аналитики и разработки на логистической платформе (складская логистика, имитационное/ресурсное моделирование, control tower). CTO поставил задачу создать платформу для 5 функций кодогенерации. Компания имеет развёрнутую AI-инфраструктуру (Qwen 3 235B, GLM 4.7, Kimi K2.5, средние/малые модели) on-premise с жёсткой сетевой изоляцией.

## Постановка задачи

Команды разработки тратят значительное время на бойлерплейт: дата-модели из protobuf, REST-эндпоинты/клиенты, скелеты сервисов. Платформа автоматизирует детерминированную генерацию где возможно и использует AI для сложных задач интерпретации.

**4 функции генерации** (Функции 3 и 5 из оригинальной постановки объединены):
1. Генерация дата-моделей из protobuf
2. Генерация REST-эндпоинтов для модели
3. Создание сервисов по PUML-спецификациям (class + sequence diagrams)
4. Генерация клиентов из OpenAPI/AsyncAPI

### 4 функции генерации — подход к реализации

#### Функция 1: Proto → Дата-модели (Детерминированная)
- **Инструмент**: `buf` (13k+ звёзд, BSR, активное сообщество)
- **Как**: buf.gen.yaml конфигурация под каждый язык, кастомные protoc-плагины для корпоративных конвенций
- **Результат**: Дата-классы/структуры с полями, типами, аннотациями
- **Языки**: Java/Kotlin (protobuf-java), Go (protoc-gen-go), Python (protobuf)

#### Функция 2: REST-эндпоинты для модели (Детерминированная + Шаблоны)
- **Инструмент**: `openapi-generator` (23k+ звёзд) с кастомными шаблонами
- **Как**: OpenAPI-спецификация → серверные заглушки с корпоративными шаблонами (Jinja2)
- **Результат**: Контроллер/хендлер + маршруты + типы запросов/ответов + стандартный middleware (авторизация, логирование, обработка ошибок)
- **Шаблоны**: Кастомизированные под фреймворк (Spring Boot для Java/Kotlin, Gin/Echo для Go, FastAPI для Python)

#### Функция 3: Сервисы из PUML-интерфейсов (AI-генерация)
- **Инструмент**: Собственный AI-пайплайн
- **Как**:
  1. PUML-текст (class + sequence диаграммы) передаётся напрямую в LLM как часть промпта — без промежуточного парсера
  2. Сборка структурированного промпта: PUML-текст + стандарты компании + шаблоны фреймворков + few-shot примеры
  3. Отправка в Qwen 3 235B / GLM 4.7 через OpenAI-compatible chat completion API
  4. Постобработка: форматирование, линтинг, валидация структуры
  5. При крупных диаграммах — разбиение на части (class diagram и sequence diagram обрабатываются отдельно)
- **Результат**: Определения интерфейсов + скелеты реализации + настройка DI + заглушки взаимодействий из sequence-диаграмм
- **Важно**: AI генерирует скелет, НЕ бизнес-логику. Разработчик заполняет логику сам.

#### Функция 4: Клиенты из OpenAPI/AsyncAPI (Детерминированная)
- **Инструмент**: `openapi-generator` (OpenAPI) + `asyncapi/generator` (AsyncAPI, 700+ звёзд)
- **Как**: Спецификация → типизированные HTTP-клиенты и async-клиенты с корпоративными конвенциями
- **Результат**: Класс клиента с методами, типы запросов/ответов, конфигурация retry/timeout. Для AsyncAPI: обработчики сообщений, привязки каналов
- **Шаблоны**: Кастомизированные под язык с учётом корпоративных стандартов HTTP-клиентов

### Аудит и логирование

Каждое событие генерации логируется во внутреннюю систему:
- Кто инициировал (пользователь/пайплайн)
- Что на входе (тип спецификации, хеш спеки)
- Что на выходе (сгенерированные файлы, язык, фреймворк)
- Какой генератор (детерминированный/AI, используемая модель)
- Длительность, статус (успех/ошибка)
- Для AI: хеш промпта, версия модели, температура

### Валидация входных данных и обработка ошибок

#### Валидация входных данных
Все входные спецификации должны быть структурированы и по возможности шаблонизированы:

| Тип входа | Валидация | Шаблон |
|---|---|---|
| Proto (.proto) | `buf lint` — проверка синтаксиса и стиля | buf.yaml с корпоративными lint-правилами |
| OpenAPI (.yaml/.json) | JSON Schema валидация по OpenAPI 3.x спецификации | Шаблон OpenAPI-файла с обязательными полями |
| AsyncAPI (.yaml/.json) | JSON Schema валидация по AsyncAPI спецификации | Шаблон AsyncAPI-файла с обязательными полями |
| PUML (.puml) | Базовая проверка: наличие `@startuml`/`@enduml`, тип диаграммы (class/sequence) | Шаблон PUML с примером структуры |

#### Стратегия обработки ошибок

**Уровни ошибок:**
1. **Ошибки валидации** (входные данные) → немедленный отказ с понятным сообщением, какое поле/строка невалидны
2. **Ошибки генерации** (детерминированный генератор упал) → логирование stderr генератора, возврат диагностики пользователю
3. **Ошибки AI** (таймаут, невалидный output, переполнение контекста) → retry с экспоненциальным backoff (макс. 3 попытки), fallback на другую модель (Qwen → GLM), при полном отказе — возврат ошибки с рекомендацией упростить PUML
4. **Ошибки постобработки** (код не прошёл линтер/форматтер) → возврат кода с предупреждением, лог конкретных ошибок линтера

**Формат ответа об ошибке:**
```json
{
  "status": "error",
  "error_code": "VALIDATION_FAILED",
  "message": "Описание ошибки на русском",
  "details": {
    "file": "order.proto",
    "line": 42,
    "rule": "FIELD_NAMES_LOWER_SNAKE_CASE"
  },
  "suggestion": "Переименуйте поле orderID в order_id"
}
```

### Деплой и инфраструктура

#### Среда
- Корпоративная облачная платформа с PaaS-сервисами
- Контейнеризация: Docker → корпоративный container registry
- Оркестрация: Kubernetes (уточнить конкретную платформу)

#### Используемые PaaS-сервисы
| Сервис | Назначение |
|---|---|
| PostgreSQL as Service | История генераций, конфигурации пользователей, метаданные шаблонов |
| Redis as Service | Брокер задач для Celery (асинхронная генерация через UI), кеш конфигураций |
| Kafka (опционально) | Публикация аудит-событий, если внутренняя система логирования потребляет из Kafka |

#### Health checks и Monitoring
- `/health` endpoint — liveness probe (приложение живо)
- `/ready` endpoint — readiness probe (подключение к PostgreSQL, Redis, AI API доступен)
- Метрики Prometheus: количество генераций, длительность, ошибки, использование AI
- Интеграция с корпоративной системой мониторинга

### Управление секретами

| Секрет | Хранение | Ротация |
|---|---|---|
| API-ключи к AI API (Qwen, GLM) | Vault / env vars платформы | По политике ИБ |
| Токены доступа к Git (для подтягивания шаблонов) | Vault / env vars платформы | По политике ИБ |
| SSO-секреты (client_id, client_secret для OAuth/OIDC) | Vault / env vars платформы | По политике ИБ |
| Пароли к PostgreSQL, Redis | Managed secrets облачной платформы | Автоматическая |

- Ни один секрет не хранится в коде, конфигах или образах
- Логируется факт обращения к секретам (кто, когда, к какому)
- Механизм горячей ротации без простоя (через Vault lease или env reload)

### Требования безопасности

- Все зависимости предзагружены во внутренние реестры (нет внешнего сетевого доступа в рантайме)
- Корпоративный SSO для аутентификации Web UI
- Управление API-ключами для CLI/CI доступа
- Сгенерированный код проходит существующие CI/CD-гейты безопасности (SAST)
- AI-промпты не содержат чувствительные бизнес-данные (санитизация входных данных при необходимости)
- Rate limiting на Core API (защита от случайных циклических вызовов из CI/CD)
- CORS-политика: разрешён только домен Web UI

## Стратегия тестирования

### Уровень 1: Unit-тесты оркестратора
- Загрузчик конфигурации: корректно читает conventions.yaml, обрабатывает отсутствие файла
- Маршрутизатор генерации: правильно определяет тип генератора по входным данным
- Валидатор входных данных: отклоняет невалидные proto/OpenAPI/PUML, принимает валидные
- Постобработка: форматирование применяется, ошибки линтера возвращаются корректно
- **Инструмент**: `pytest`, цель покрытия — 80%+ для core-модулей

### Уровень 2: Интеграционные тесты генерации
Для каждой функции × каждого языка:
1. Взять эталонную спецификацию (proto / OpenAPI / AsyncAPI / PUML)
2. Запустить генерацию
3. Проверить, что результат:
   - Компилируется / парсится на целевом языке
   - Проходит корпоративные линтеры
   - Содержит обязательные элементы (импорты internal-библиотек, аннотации и т.д.)
- **Инструмент**: `pytest` + Docker-контейнеры с компиляторами (javac, go build, python -m py_compile)

### Уровень 3: Golden-файлы (регрессия)
- Для каждой комбинации (функция × язык × фреймворк) хранить эталонный результат генерации
- При изменении шаблона — автоматическое сравнение нового результата с эталоном
- Различие → тест падает → разработчик ревьюит изменение и обновляет эталон
- **Инструмент**: `pytest` + snapshot-тестирование (`syrupy` или аналог)

### Уровень 4: AI-генерация (Функция 3)
AI-генерация недетерминированна, поэтому тестирование отличается:
- **Структурные проверки**: результат содержит ожидаемые интерфейсы/классы из PUML
- **Компилируемость**: сгенерированный код компилируется
- **Линтинг**: проходит корпоративные линтеры
- **Ручная оценка (на пилоте)**: разработчики пилотных команд оценивают качество по шкале 1-5
- **Метрики**: % компилируемых результатов, % прошедших линтер, средняя оценка разработчиков

### Уровень 5: E2E-тесты
- CLI: `aicreator generate --from proto --to kotlin --spec order.proto` → файлы на диске → компилируются
- API: POST `/api/generate` → JSON-ответ с файлами → валидны
- Web UI: Selenium/Playwright smoke-тест загрузки спеки и получения результата

### CI/CD платформы
- Pre-commit: линтинг и типизация (ruff, mypy)
- PR: unit + интеграционные + golden-файлы
- Nightly: полный E2E + AI-генерация с проверкой компилируемости
