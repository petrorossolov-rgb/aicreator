# План презентации AICreator для CTO

## Мета-информация
- **Формат**: ~20 слайдов, ~25 минут + 15 минут Q&A
- **Аудитория**: CTO, возможно технические директора стримов
- **Цель**: Представить техническое решение, получить одобрение на запуск пилота, согласовать ресурсы

---

## Слайд 1: Титульный
- Название: AICreator — Платформа AI-генерации кода
- Подзаголовок: Техническое решение по задаче ускорения разработки на логистической платформе
- Автор, дата, статус: проект решения

## Слайд 2: Предлагаемое решение — обзор
- AICreator: платформа, которая генерирует код из формальных спецификаций
- 4 функции генерации (кратко, по одной строке каждая)
- Ключевой принцип: **детерминированная генерация** где возможно, **AI** только для сложных задач
- Генерируется скелет + стандартные обвязки, НЕ бизнес-логика

## Слайд 3: 4 функции генерации (детально)
Таблица:

| # | Функция | Вход | Выход | Генератор |
|---|---|---|---|---|
| 1 | Дата-модели | .proto | Классы/структуры | buf (детерм.) |
| 2 | REST-эндпоинты | OpenAPI | Контроллеры + middleware | openapi-generator (детерм.) |
| 3 | Сервисы из PUML | .puml | Интерфейсы + скелеты | AI (Qwen 3 235B) |
| 4 | Клиенты | OpenAPI/AsyncAPI | HTTP/async-клиенты | openapi-gen + asyncapi-gen (детерм.) |

Акцент: 3 из 4 функций — детерминированные, предсказуемые, без рисков AI

## Слайд 4: Почему не «всё на AI»?
- Enterprise-приоритеты: безопасность, качество, предсказуемость
- Детерминированные генераторы: один вход → всегда один и тот же выход
- AI используется только для Функции 3 (PUML → сервисы), где нет детерминированной альтернативы
- 75% генерации — полностью предсказуемо и контролируемо
- AI-часть тоже ограничена: скелет, не бизнес-логика

## Слайд 5: Три интерфейса — один для каждого пользователя
- **CLI** — для разработчиков (терминал, IDE)
- **Web UI** — для системных аналитиков (загрузка спеки → получение кода)
- **CI/CD-плагин** — автоматическая генерация при изменении спецификаций
- Диаграмма: три пользователя → три интерфейса → одно ядро

## Слайд 6: Архитектура — C4 Context

  - Корпоративный SSO

## Слайд 9: Использование AI-инфраструктуры компании
- Используем уже развёрнутые модели: Qwen 3 235B (основная), GLM 4.7 (тестовая)
- OpenAI-compatible API — нативная интеграция через litellm
- Model-agnostic: смена модели через конфигурацию, без изменения кода
- Не нужны новые GPU / деплой моделей — переиспользуем существующее

## Слайд 10: Корпоративные стандарты в генерации
- Шаблоны и стандарты — в отдельном Git-репозитории
- Версионирование через git tags, обновление через PR
- Включают: код-стайл, линтеры, архитектурные паттерны, internal-библиотеки
- Генерируемый код сразу соответствует корпоративным требованиям
- Показать структуру corp-codegen-templates/

## Слайд 11: Сценарий 1 — Разработчик генерирует через CLI
- Упрощённая версия seq-cli-generation.puml
- Одна команда → файлы в проекте → соответствуют стандартам
- Демо-пример: `aicreator generate --from proto --to kotlin --spec order.proto`

## Слайд 12: Сценарий 2 — Аналитик генерирует через Web UI
- Упрощённая версия seq-ui-puml-generation.puml
- Загрузка PUML → выбор языка → получение скелета сервиса
- Аналитик может сразу видеть, как его спецификация превращается в код

## Слайд 13: Сценарий 3 — Автогенерация в CI/CD
- Упрощённая версия seq-cicd-generation.puml
- Обновил OpenAPI → CI автоматически обновил клиенты → MR
- Цель: zero-touch генерация

## Слайд 14: Безопасность и compliance
- Все зависимости — во внутренних реестрах, нет внешнего доступа в рантайме
- Аутентификация: SSO (Web UI), API-ключи (CLI/CI)
- Секреты: Vault / managed secrets платформы, ротация
- Аудит: каждая генерация логируется (кто, что, когда, какая модель)
- Сгенерированный код проходит существующие SAST-гейты
- AI-промпты не содержат ПД и коммерческую тайну

## Слайд 15: Качество — стратегия тестирования
- 5 уровней тестирования:
  1. Unit-тесты оркестратора (pytest, 80%+ покрытие)
  2. Интеграционные тесты: спека → генерация → компиляция
  3. Golden-файлы: регрессия при изменении шаблонов
  4. AI-метрики: компилируемость, линтинг, ручная оценка разработчиков
  5. E2E: полный цикл через CLI/API/UI
- Акцент: сгенерированный код гарантированно компилируется и проходит линтеры

## Слайд 16: Обработка ошибок
- 4 уровня ошибок: валидация → генерация → AI → постобработка
- Структурированные ответы с конкретным описанием проблемы и рекомендацией
- AI: retry с экспоненциальным backoff, fallback на альтернативную модель
- При полном отказе AI — функции 1, 2, 4 продолжают работать (детерминированные)

## Слайд 18: Фазы — ключевые deliverables

|---|---|---|

## Слайд 19: Пилот — масштаб и метрики
- 1-2 команды: идеально одна Java/Kotlin + одна Go
- У пилотных команд уже есть proto/OpenAPI спецификации
- Метрики успеха:
  - Сэкономленное время (часы/неделя на команду)
  - Качество: % сгенерированного кода, прошедшего code review без правок
  - Адопция: % разработчиков команды, использующих инструмент регулярно
  - NPS: удовлетворённость пользователей

## Слайд 20: Необходимые ресурсы
- **Команда**: 2-3 Python-бэкенд + 1 React-фронтенд (с Фазы 4)
- **Инфраструктура**: PostgreSQL, Redis (as Service — уже есть), container registry, CI/CD
- **AI**: переиспользуем существующие Qwen 3 235B / GLM 4.7 — новые GPU не нужны
- **Вопросы для решения**: состав команды, выбор пилотных команд, уточнение CI/CD / Vault / K8s

## Слайд 21: Риски и митигация
| Риск | Вероятность | Митигация |
|---|---|---|
| Качество AI-генерации ниже ожиданий | Средняя | A/B тестирование моделей, fallback, только скелет без бизнес-логики |
| Команды не примут инструмент | Средняя | Гибрид CLI+UI, пилот с мотивированными командами, сбор фидбэка |
| OSS-зависимости недоступны в изол. сети | Низкая | Предзагрузка в Фазе 0, проверка до старта разработки |
| Переполнение контекста LLM на крупных PUML | Низкая | Разбиение диаграмм, мониторинг размера промпта |

## Слайд 22: Что мы получим
- **Ускорение разработки**: автоматизация 60-70% бойлерплейта
- **Стандартизация**: весь генерируемый код соответствует корпоративным требованиям
- **Масштабируемость**: после пилота — тиражирование на все стримы
- **Использование AI-инфраструктуры**: практическое применение уже развёрнутых моделей
- **Конкурентное преимущество**: ускорение time-to-market для логистических сервисов

## Рекомендации по оформлению

### Стиль
- Минималистичный корпоративный дизайн, без лишних украшений
- Тёмная или светлая тема — под корпоративный стиль
- Максимум 5-7 строк текста на слайде, остальное — схемы и таблицы

### Визуализация
- C4-диаграммы: отрендерить из PUML-файлов, вставить как изображения
- Sequence-диаграммы: упрощённые версии (не все alt-блоки, только happy path)
- Ганттоподобная визуализация фаз — горизонтальная шкала
- Таблицы: минимум столбцов, крупный шрифт
