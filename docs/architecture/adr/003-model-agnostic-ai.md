# ADR-003: Model-agnostic AI-слой через litellm

## Статус
Принято

## Контекст
Компания имеет развёрнутую AI-инфраструктуру с несколькими LLM:
- **Qwen 3 235B** (131K токенов) — основная модель, доступна по API
- **GLM 4.7** (147K токенов) — размышляющая модель, доступна по API
- Средние и малые модели для вспомогательных задач

Все модели предоставляют **OpenAI-compatible API** (chat completion, completion, embedding).

Рынок LLM быстро развивается. Модели, которые лучшие сегодня, могут уступить через 3-6 месяцев. Решение не должно быть привязано к конкретной модели.

## Рассмотренные варианты

### A) Прямая интеграция с API каждой модели
- (+) Минимум зависимостей
- (−) Жёсткая привязка к конкретному API
- (−) Смена модели = изменение кода

### B) Собственная абстракция
- (+) Полный контроль
- (−) Затраты на разработку и поддержку
- (−) Изобретение велосипеда

### C) litellm (выбран)
- (+) 16k+ звёзд, активное сообщество, регулярные обновления
- (+) OpenAI-compatible из коробки
- (+) Смена модели через конфигурацию, без изменения кода
- (+) Встроенные retry, fallback, load balancing
- (+) Поддержка 100+ провайдеров
- (−) Дополнительная зависимость

## Решение
Использовать **litellm** как model-agnostic слой для взаимодействия с LLM.

```python
from litellm import completion

response = completion(
    model="openai/qwen-3-235b",   # смена модели — одна строка в конфиге
    api_base="https://internal-ai-api.corp.local",
    messages=[...],
    temperature=0.1,
)
```

Конфигурация моделей (endpoint, имя, температура, макс. токены) — в YAML-файле, не в коде. A/B-тестирование между Qwen и GLM — через конфигурацию.

Fallback-стратегия: Qwen 3 235B → GLM 4.7 → ошибка с рекомендацией.

## Последствия
- Легко добавлять и тестировать новые модели без изменения кода
- A/B-тестирование на Фазе 3 не требует разработки — только конфигурация
- litellm должен быть загружен во внутренний PyPI registry
- Нужно следить за обновлениями litellm для совместимости с новыми API
