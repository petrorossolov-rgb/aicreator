# ADR-002: Детерминированные шаблоны + AI для сложных случаев

## Статус
Принято

## Контекст
Платформа выполняет 4 функции кодогенерации. Для части из них (proto → модели, OpenAPI → эндпоинты/клиенты) существуют зрелые детерминированные инструменты (buf, openapi-generator). Для генерации сервисов из PUML-диаграмм детерминированного решения нет — требуется «понимание» спецификации.

В крупной enterprise-компании приоритеты: **безопасность, качество, предсказуемость, минимизация рисков**. Недетерминированность LLM (один и тот же вход → разный выход) является серьёзным риском для промышленной кодогенерации.

## Рассмотренные варианты

### A) AI-first — всё через LLM
- (+) Максимальная гибкость, один универсальный генератор
- (−) Недетерминированность: разный результат при повторных запусках
- (−) Высокие затраты GPU на каждую генерацию
- (−) Сложность контроля качества и code review
- (−) Зависимость от доступности AI API

### B) Шаблоны + AI для сложных случаев (выбран)
- (+) 75% генерации — предсказуемый, стабильный результат
- (+) AI используется только там, где шаблоны не справляются
- (+) Меньше затраты GPU
- (+) Детерминированные генераторы работают без сети к AI API
- (−) Требует разработки и поддержки кастомных шаблонов для каждого языка/фреймворка

### C) AI + постобработка — LLM генерирует, валидаторы проверяют
- (+) Гибко, как вариант A
- (−) Те же проблемы недетерминированности
- (−) Валидаторы ловят синтаксические ошибки, но не логические

## Решение
Разделение функций по типу генератора:

| Функция | Генератор | Обоснование |
|---|---|---|
| 1. Proto → Дата-модели | `buf` (детерминированный) | Прямая трансляция, нет интерпретации |
| 2. REST-эндпоинты | `openapi-generator` (детерминированный) | Стандартная генерация по OpenAPI-спеке |
| 3. Сервисы из PUML | AI (Qwen 3 235B / GLM 4.7) | Требует интерпретации диаграмм |
| 4. Клиенты OpenAPI/AsyncAPI | `openapi-generator` + `asyncapi-generator` (детерминированный) | Прямая трансляция |

AI генерирует **скелет + стандартные обвязки**, но **не бизнес-логику**. Разработчик заполняет логику самостоятельно.

## Последствия
- Кастомные шаблоны (Jinja2) для детерминированных генераторов требуют начальных инвестиций и поддержки
- Качество AI-генерации (Функция 3) нужно измерять и контролировать отдельно
- При недоступности AI API функции 1, 2, 4 продолжают работать
- Постепенное расширение: по мере улучшения моделей можно перевести больше задач на AI
