# ADR-005: Критерии выбора OSS-зависимостей (700+ звёзд)

## Статус
Принято

## Контекст
AICreator — enterprise-продукт для крупной ритейл-компании с жёсткими требованиями к безопасности, надёжности и поддерживаемости. Все зависимости должны работать в изолированной корпоративной сети и предзагружаться во внутренние реестры.

Выбор ненадёжной OSS-зависимости несёт риски:
- Прекращение поддержки (orphaned project)
- Уязвимости без патчей
- Отсутствие community support при проблемах
- Несовместимость с новыми версиями языков/платформ

## Рассмотренные варианты

### A) Без формальных критериев
- (+) Максимальная свобода выбора
- (−) Риск попадания незрелых зависимостей в продукт
- (−) Нет объективного фильтра при обсуждении с командой

### B) Формальные критерии (выбран)
- (+) Объективный фильтр, снижающий риски
- (+) Упрощает принятие решений и code review
- (−) Ограничивает выбор для нишевых задач

## Решение
Каждая OSS-зависимость AICreator должна соответствовать **всем трём критериям**:

1. **700+ звёзд на GitHub** — индикатор зрелости и доверия сообщества
2. **Активное сообщество** — регулярные коммиты, ответы на issues, PR от внешних контрибьюторов
3. **Регулярные обновления** — релизы не реже чем раз в 6 месяцев

### Процесс исключений
Если для конкретной задачи нет OSS-решения, соответствующего критериям:
1. Рассмотреть альтернативный подход (как с PUML — вместо парсера отдали текст напрямую в LLM)
2. Написать минимальное собственное решение (< 500 строк)
3. В крайнем случае — обоснование исключения с оценкой рисков, утверждение архитектором

## Последствия
- Все зависимости проходят единый фильтр качества
- Снижен риск abandoned dependencies
- Для нишевых задач может потребоваться больше собственного кода
- Критерии нужно пересматривать ежегодно (порог 700 может измениться)
