# ADR-004: Передача PUML напрямую в LLM без промежуточного парсера

## Статус
Принято

## Контекст
Функция 3 (генерация сервисов из PUML) требует обработки PlantUML-диаграмм (class + sequence). Необходимо решить, как извлекать информацию из PUML для генерации кода.

Корпоративный критерий OSS: 700+ звёзд на GitHub, активное сообщество, регулярные обновления.

## Рассмотренные варианты

### A) plantuml-parser (JS, ~150 звёзд)
- (+) Структурированный вывод (AST)
- (−) **Не проходит корпоративный критерий** (150 < 700 звёзд)
- (−) JavaScript-зависимость в Python-проекте
- (−) Поддержка ограничена — малое сообщество

### B) PlantUML JAR → XMI экспорт
- (+) Официальный инструмент PlantUML (10k+ звёзд)
- (+) XMI — стандартный формат, легко парсить
- (−) Дополнительная JVM-зависимость
- (−) XMI-экспорт поддерживает не все типы диаграмм
- (−) Лишний шаг трансформации

### C) PUML-текст напрямую в LLM (выбран)
- (+) Нулевые дополнительные зависимости
- (+) LLM (Qwen 3 235B, GLM 4.7) отлично понимают PlantUML-синтаксис
- (+) Сохраняется полный контекст диаграммы (комментарии, заметки, аннотации)
- (−) Риск переполнения контекста на крупных диаграммах

## Решение
Передавать **PUML-текст как есть** в промпт LLM. Модель самостоятельно извлекает интерфейсы, методы, зависимости и потоки взаимодействия.

**Митигация риска переполнения контекста:**
- Для крупных диаграмм: разбиение на class diagram + sequence diagram и обработка отдельными запросами
- Мониторинг размера промпта (предупреждение при > 80% контекстного окна)
- Приоритет информации: сначала class diagram (интерфейсы), затем sequence diagram (взаимодействия)

## Последствия
- Одной зависимостью меньше
- Качество генерации зависит от способности модели понимать PUML (проверяется на Фазе 3)
- Нужно разработать стратегию разбиения крупных диаграмм
- При замене LLM-провайдера нужно проверять способность новой модели интерпретировать PUML
